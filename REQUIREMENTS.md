# 家計簿アプリ 要件定義書

- **バージョン**: v0.1
- **作成日**: 2026-05-26
- **対象**: 個人開発の Web 家計簿アプリ

---

## 1. 概要

個人利用を前提とした Web ベースの家計簿アプリ。PC・スマートフォン両対応（レスポンシブ）で、クラウド DB によって複数端末からデータを同期できる。

### 1.1 ターゲットユーザー
- アプリ作成者本人（1 ユーザー）
- 家族間共有や一般公開は対象外（フェーズ 2 以降で検討）

### 1.2 目的
- 日々の収支を簡単に記録し、月次でカテゴリ別の支出傾向を把握する
- カテゴリ別予算を立て、超過を視覚的に把握する
- 複数の口座（現金・銀行・カード）の残高を一元管理する

---

## 2. 機能要件

| # | 機能 | 概要 | 優先度 |
|---|---|---|---|
| F1 | 認証 | メール+パスワードでのログイン／ログアウト／パスワードリセット（Firebase Auth） | 必須 |
| F2 | 収入・支出入力 | 日付・金額・カテゴリ・口座・メモを手入力。一覧・編集・削除が可能 | 必須 |
| F3 | カテゴリ管理 | 既定カテゴリを用意しつつ、ユーザーが追加・編集・削除できる | 必須 |
| F4 | 集計・グラフ | 月次サマリ表示、カテゴリ別の円グラフ・棒グラフ表示 | 必須 |
| F5 | 予算管理 | カテゴリごとに月次予算を設定。超過時は画面上で赤字表示＋バナー表示 | 必須 |
| F6 | 定期・繰り返し取引 | 家賃・サブスク等を月次自動計上（テンプレート登録方式） | 必須 |
| F7 | 複数口座・資産管理 | 口座（現金・銀行・カード等）の追加。口座間の振替。各口座残高表示 | 必須 |
| F8 | CSV インポート／エクスポート | 全取引のエクスポート、規定フォーマットでのインポート | 必須 |

### 2.1 機能詳細メモ

- **F2 取引入力**
  - 必須項目: 日付・金額・収支区分・カテゴリ・口座
  - 任意項目: メモ・タグ
  - 入力後、月次サマリと予算消化率は即座に更新される
- **F5 予算アラート**
  - アラートはアプリ内表示のみ（メール通知・プッシュ通知は対象外）
  - 月次予算が 80% 到達でバナー表示（黄）、100% 超過で赤バナー
- **F6 定期取引**
  - MVP では Cloud Functions のバッチ処理は使わず、アプリ起動時に「未計上の定期取引を生成」する方式で実装
- **F7 口座振替**
  - 1 つの振替操作で「出金側の取引」と「入金側の取引」の 2 レコードを生成
  - 振替は損益（収支）には含めない
- **F8 CSV**
  - エクスポート列: 日付, 収支区分, 金額, カテゴリ, 口座, メモ, タグ
  - インポートは同一フォーマットを想定。重複検知は MVP では行わない

## 3. 非機能要件

| 項目 | 内容 |
|---|---|
| 対応ブラウザ | Chrome / Safari / Edge の最新版 |
| レスポンシブ | スマホ縦持ち（375px〜）／タブレット／PC（1280px〜）対応 |
| パフォーマンス | 取引一覧 100 件で 1 秒以内表示 |
| セキュリティ | Firestore Security Rules で `userId == request.auth.uid` の取引のみ読み書き可 |
| 通貨 | 日本円のみ |
| 言語 | 日本語のみ |
| アクセシビリティ | デジタル庁デザインシステム（DADS）準拠を目安。コントラスト比 4.5:1 以上、タップ領域 44px 以上 |
| バックアップ | Firestore の標準スナップショットに依存。ユーザー側は CSV エクスポートで対応 |

## 5. データモデル（概要）

```
users/{userId}
  ├── accounts/{accountId}
  │     - name: string            口座名（例: 三井住友銀行）
  │     - type: 'cash' | 'bank' | 'card' | 'other'
  │     - initialBalance: number  初期残高
  │     - createdAt: timestamp
  │
  ├── categories/{categoryId}
  │     - name: string
  │     - color: string           表示用カラーコード
  │     - kind: 'income' | 'expense'
  │     - isDefault: boolean
  │
  ├── budgets/{budgetId}
  │     - yearMonth: string       例: "2026-05"
  │     - categoryId: string
  │     - amount: number
  │
  ├── transactions/{txId}
  │     - date: timestamp
  │     - amount: number
  │     - kind: 'income' | 'expense' | 'transfer'
  │     - categoryId: string
  │     - accountId: string
  │     - counterAccountId: string?   振替時のみ
  │     - memo: string
  │     - tags: string[]
  │     - recurringId: string?       定期取引から生成された場合
  │     - createdAt: timestamp
  │
  └── recurring/{recurringId}
        - name: string
        - amount: number
        - kind: 'income' | 'expense'
        - categoryId: string
        - accountId: string
        - dayOfMonth: number         毎月の計上日
        - nextRunDate: timestamp
        - active: boolean
```

## 6. 画面構成（仮）

1. **ログイン画面** — メール+パスワード入力
2. **ダッシュボード** — 今月の収支サマリ、予算消化率、最近の取引
3. **取引一覧** — フィルタ（期間・カテゴリ・口座）、編集・削除
4. **取引入力（モーダル or 専用ページ）** — クイック入力
5. **カテゴリ別グラフ** — 月次の円グラフ・棒グラフ
6. **予算設定画面** — カテゴリごとの月次予算
7. **口座一覧／残高画面** — 各口座の現在残高、振替操作
8. **定期取引設定画面** — テンプレートの追加・編集
9. **CSV インポート／エクスポート画面**
10. **設定画面** — カテゴリ管理、アカウント情報、ログアウト

## 8. リスク・懸念点

| リスク | 影響 | 対応方針 |
|---|---|---|
| 定期取引のバッチ実装 | 工数増（Cloud Functions 必要） | MVP はアプリ起動時生成方式で簡略化 |
| 口座振替のデータ整合性 | 1 操作で 2 レコード → 不整合の余地 | Firestore のバッチ書き込み（`writeBatch`）で原子的に処理 |
| Firestore コスト | 個人利用想定だが読み取り回数次第で課金発生 | 月次サマリはキャッシュ。クエリにインデックスを設計 |
| スマホでのグラフ表示 | Recharts は小画面でラベルが崩れやすい | レスポンシブ確認を必ず実施。必要なら凡例を縦並びに |
| デザインシステム準拠の工数 | 完全準拠は重い | MVP ではコントラスト・タップ領域・フォーカス可視化のみを優先 |
